# 轻小说 TXT 整理与解析格式说明

本文档定义项目当前使用的轻小说文本整理格式，目标是同时服务两类工作：

- 文本整理人员：按照统一规则把轻小说原文整理成可导入的 TXT。
- 代码维护人员：根据固定字段解析文本，生成展示内容与 TTS 播放队列。

核心原则：

- 所有可进入 TTS 的文本必须是“可朗读文本”。
- 所有不能稳定进入 TTS 的符号行、分隔线、装饰行，不作为句子传入 TTS。
- 文本格式使用显式字段，不依赖“第几行默认是什么”的猜测逻辑。
- 最终落盘格式只允许 `ja:` / `zh:` / `word:` / `pause:` 四种前缀。
- 历史遗留格式（如 `jp:`、`word1:`、第三行混合词汇串）只可作为迁移输入，不可继续保留到最终文件中。

## 1. 基本结构

每个内容块代表一个阅读单元，块与块之间使用一个或多个空行分隔。

标准内容块格式如下：

```txt
ja: 隣[となり]で本[ほん]から顔[かお]も上[あ]げずに答[こた]えたのは、文芸部[ぶんげいぶ]副部長[ふくぶちょう]の小[こ]鞠[まり]知[ち]花[か]。
zh: 在旁边连头都没从书上抬起来回答的，是文艺部副部长小鞠知花。
word: 隣 | となり | 旁边
word: 顔も上げずに | かおもあげずに | 连头都不抬
word: 答える | こたえる | 回答
word: 文芸部 | ぶんげいぶ | 文艺部
word: 副部長 | ふくぶちょう | 副部长
```

如果原文某一行本身不是可朗读句子，而只是停顿、分隔、强调符号或装饰行，则单独写成：

```txt
pause: 500
```

含义是暂停 500 毫秒，也就是 0.5 秒。`pause` 不走 TTS，只生成静音停顿。

## 2. 字段定义

当前只定义以下 4 种字段：

- `ja:` 日文原句。
- `zh:` 中文译句。
- `word:` 词汇解释行。
- `pause:` 静音停顿，单位为毫秒。

除这 4 种字段外，TXT 中不应出现其他前缀。若后续要扩展新字段，应先更新本说明和解析代码。

## 3. 各字段规则

### 3.1 `ja:`

用途：保存当前阅读单元的日文原句。

规则：

- 每个句子块最多 1 行 `ja:`。
- `ja:` 后面必须是可阅读、可朗读的日文正文。
- `ja:` 与 `zh:` 必须逐句一一对应。
- 允许使用 `漢字[かな]` 形式保留注音。
- `ja:` 是该句子块的主字段，没有 `ja:` 的普通句子块应视为不完整。
- 最终文件统一使用 `ja:`，不得继续保留历史 `jp:`。

注音精修要求：

- 注音必须使用标准读音，不可凭感觉乱补。
- 优先给“人名 / 地名 / 作品名 / 难读词 / 容易误读词”补注音。
- 优先使用整词注音，避免把本来应整体读的词硬拆成逐字注音。
- 不要对纯假名、助词、普通片假名外来语做无意义注音。
- 已有注音若错误，必须改正后再落盘。
- 同一专名在全书中必须使用同一读音写法。

示例：

```txt
ja: 今日[きょう]は図書[としょ]委員[いいん]の手伝[てつだ]いで、二人[ふたり]で受付[うけつけ]をしているのだ。
```

### 3.2 `zh:`

用途：保存与 `ja:` 对应的中文译句。

规则：

- 每个句子块最多 1 行 `zh:`。
- `zh:` 应与当前 `ja:` 严格一一对应。
- `zh:` 可以为空缺，但推荐始终提供，便于展示和双语播放。
- `zh:` 必须只写译文，不得夹带词汇表、注释串、自然语言式词条说明。
- 如果原文切成两句，则中文也应切成两句，不得把多句内容糊成一条 `zh:`。

逐句翻译精修要求：

- 译文要忠实原意，同时保证中文自然顺口。
- 保留原句语气、人物口吻、敬体/简体差异带来的语感。
- 对话句保留合适的引号与停顿，不要把对白译成旁白腔。
- 不把词汇解释硬塞进译文中。
- 不要为了“看起来完整”而擅自补写原文没有的信息。

示例：

```txt
zh: 今天来帮忙图书委员的工作，我们两人负责接待。
```

### 3.3 `word:`

用途：保存当前句子中需要重点学习的词汇。

格式：

```txt
word: 表记 | 读音 | 释义
```

规则：

- 一个句子块可以有 0 到多行 `word:`。
- 每行 `word:` 固定拆成 3 段，用竖线 `|` 分隔。
- 最终顺序固定为：`表记 | 读音 | 释义`。
- `表记` 用词条本身，例如 `図書委員`。
- `读音` 用假名或注音，例如 `としょいいん`。
- `释义` 用中文简洁说明，例如 `图书委员`。
- 竖线两侧允许有空格，解析时应自动 `trim()`。
- 一条词条只占一行，禁止把多个词条塞进同一行 `word:`。

示例：

```txt
word: 図書委員 | としょいいん | 图书委员
word: 手伝い | てつだい | 帮忙
word: 受付 | うけつけ | 接待
```

不推荐写法：

```txt
word: 図書委員(としょいいん): 图书委员
```

原因：这种自然语言式写法不利于稳定解析，也不利于后续独立展示“表记 / 读音 / 释义”。

#### 3.3.1 历史遗留 `word1:` / `word2:` / 旧词汇串的提取与改写规则

历史文件中可能出现以下旧格式：

- `word1:` / `word2:` / `word3:` ... 这样的序号字段。
- 无前缀第三行词汇串，例如：`図書委員 (としょいいん): 图书委员, 手伝い (てつだい): 帮忙`
- 误写进 `zh:` 的词汇解释串。
- 一行里混写多个词条。

这些格式都只可作为迁移输入，最终输出必须统一改写成多行 `word:`。

提取规则：

1. `word1:` / `word2:` / `word3:` 等序号前缀在抽取时一律视为普通词条来源，去掉数字后统一改写为 `word:`。
2. 旧格式中一行含多个词条时，必须拆成多行 `word:`，一个词条一行。
3. 优先从当前句子的 `ja:` 中抽取确实出现、且有学习价值的词条，不把整句翻译、备注说明或人物评价误当词条。
4. 词条默认按其在当前 `ja:` 中“第一次出现的位置”升序排序。
5. 如果两个候选词条起始位置相同，优先保留更长、更完整的词组；若仍无法区分，再保持原来源顺序。
6. 同一块内相同 `表记 + 读音 + 释义` 的词条应去重。
7. 同一表记若存在多个可能读音或义项，只保留与当前句义最匹配的一条。
8. 词条应尽量保持与句内出现形式一致；若明显需要用词典形解释（例如活用词），可改成词典形，但读音和释义必须与当前句义一致。
9. 人名、地名、作品名等专名只有在该句确有学习价值时才加入 `word:`；不要机械地每句都重复收录。
10. `zh:` 中不得残留词汇解释串；迁移后此类信息必须全部移入独立 `word:` 行，或在无必要时删除。

推荐策略：

- 每句优先保留 0～6 条高价值词汇。
- 优先保留难词、固定搭配、双关点、常见但不易直译的表达。
- 避免把“显然一看就懂”的基础助词、语尾、单个标点加入 `word:`。

### 3.4 `pause:`

用途：表示静音停顿，不进入 TTS。

格式：

```txt
pause: 500
```

规则：

- `pause:` 后必须是正整数，单位毫秒。
- 推荐默认值为 `500`。
- `pause:` 可单独作为一个块存在。
- `pause:` 不应和 `ja:` / `zh:` / `word:` 混在同一个块中。

## 4. 原文整理规则

整理轻小说原文时，按以下规则处理：

### 4.1 正常句子

原文中正常可朗读的日文句子，整理为：

```txt
ja: ...
zh: ...
word: ...
```

其中 `word:` 行可按需要补充，多行都可以。

### 4.2 分隔线、装饰线、奇形怪状符号行

如果原文某一整行属于以下情况：

- 只有破折号、波浪线、星号、圆点、方块等装饰符号。
- 只有分隔线意义，没有正文意义。
- 只有单个或多个重复符号，传入 TTS 没有朗读价值。
- 很可能导致 TTS 不稳定、报错、空音频或异常停顿。

则统一替换为：

```txt
pause: 500
```

典型例子包括但不限于：

```txt
──
......
……
***
◇
◆
□
■
【】
（空装饰行）
```

都应整理为：

```txt
pause: 500
```

### 4.3 单独一行的感叹号、问号、省略号等

如果原文单独一行只有这些内容：

- `！`
- `？`
- `!?`
- `……`
- `!!!`
- `???`
- `——`

且该行本质上只是节奏、停顿、转场，而不是需要单独朗读的正文，则统一替换为：

```txt
pause: 500
```

处理原则：

- 优先保证 TTS 稳定。
- 不把无实际朗读意义的符号行送进 TTS。
- 由静音停顿承担节奏表现。

### 4.4 句中标点正常保留

如果标点属于正常句子的一部分，而不是单独占一整行，则保留在 `ja:` 或 `zh:` 句子中，不需要替换。

例如以下情况是正常的：

```txt
ja: えっ？
zh: 咦？
```

这里 `？` 属于句子内容的一部分，不应改成 `pause: 500`。

### 4.5 逐句翻译 / 注音精修流程

整理时建议严格按以下顺序执行：

1. 先把原文切成“可以独立朗读”的最小句子块。
2. 每个句子块只保留一条 `ja:` 和一条 `zh:`，做到逐句对齐。
3. 检查并修正 `ja:` 里的注音，确保人名、难词、易误读词的读法准确。
4. 再从该句中抽取 `word:`，不要先堆词汇再反推句子。
5. 最后检查是否存在纯符号块；若有，改为 `pause:`。

额外要求：

- 不得把两句日文合成一句中文，也不得把一句日文拆成多条中文说明。
- 不得把词汇解释继续留在中文行里充当“补充翻译”。
- 同一人物、同一专名、同一固定译法必须在全卷保持一致。
- 对标题、目录、Intermission、小节标题等，也应按同样标准处理：有正文意义则用 `ja:` / `zh:`，纯装饰则用 `pause:`。

### 4.6 《負けヒロインが多すぎる！》人物姓名标准读音

整理《負けヒロインが多すぎる！》时，涉及以下人物姓名时，统一按本表读音处理；若原文未显式标注注音，也应以此为准：

- 温水和彦｜ぬくみず かずひこ
- 八奈見杏菜｜やなみ あんな
- 焼塩檸檬｜やきしお れもん
- 小鞠知花｜こまり ちか
- 温水佳樹｜ぬくみず かじゅ
- 月之木古都｜つきのき こと
- 玉木慎太郎｜たまき しんたろう
- 白玉リコ｜しらたま リコ
- 志喜屋夢子｜しきや ゆめこ
- 放虎原ひばり｜ほうこばる ひばり
- 馬剃天愛星｜ばそり てぃあら
- 桜井弘人｜さくらい ひろと
- 綾野光希｜あやの みつき
- 朝雲千早｜あさぐも ちはや
- 袴田草介｜はかまだ そうすけ

- 小抜小夜｜こぬき さよ
- 権藤アサミ｜ごんどう アサミ

执行要求：

- 同一人物在全书中不得混用不同读法。
- 若句中使用了该人物姓名并添加注音，注音必须与本表一致。
- 若词汇行收录人物姓名，`word:` 的 `读音` 也必须与本表一致。
- 若遇到本表未列出、但在本作中反复出现的重要人物，新增前应先确认官方或权威来源读音，再统一补入本规范。

## 5. 推荐块结构

推荐每个句子块严格按以下顺序书写：

```txt
ja: ...
zh: ...
word: ...
word: ...
word: ...
```

推荐规则：

- `ja:` 放第一行。
- `zh:` 放第二行。
- 所有 `word:` 放在后面。
- 块与块之间用空行分开。
- `pause:` 单独成块。

推荐示例：

```txt
ja: ──新生[しんせい]文芸部[ぶんげいぶ]の誕[たん]生[たんじょう]から一カ月[いっかげつ]。
zh: ──新生文艺部诞生至今已有一个月。
word: 新生 | しんせい | 新生
word: 誕生 | たんじょう | 诞生

pause: 500

ja: 今日[きょう]は図書[としょ]委員[いいん]の手伝[てつだ]いで、二人[ふたり]で受付[うけつけ]をしているのだ。
zh: 今天来帮忙图书委员的工作，我们两人负责接待。
word: 図書委員 | としょいいん | 图书委员
word: 手伝い | てつだい | 帮忙
word: 受付 | うけつけ | 接待
```

## 6. 代码侧配合要求

为了让该格式稳定落地，解析与 TTS 代码需要同时配合。

### 6.1 解析器要求

解析器应满足以下规则：

- 识别 `ja:`、`zh:`、`word:`、`pause:` 4 种前缀。
- 块之间按空行分隔。
- 一个普通句子块中：
  - 最多 1 个 `ja:`
  - 最多 1 个 `zh:`
  - 允许多个 `word:`
- `pause:` 必须单独成块。
- `word:` 必须按 `表记 | 读音 | 释义` 拆成 3 段。
- 遇到未知前缀时，建议报出行号和错误原因。
- 遇到格式不完整的 `word:` 行时，建议跳过并记录警告，或直接判为格式错误。
- 若需要兼容历史文件，应在“导入/迁移阶段”先把 `jp:` / `word1:` / 旧词汇串归一化，再交给正式解析器；正式解析器不应长期依赖旧格式。

推荐解析后的数据结构：

```ts
type ParsedBlock =
  | {
      type: 'entry';
      ja: string;
      zh?: string;
      words: Array<{
        surface: string;
        reading: string;
        gloss: string;
      }>;
    }
  | {
      type: 'pause';
      durationMs: number;
    };
```

### 6.2 TTS 生成要求

TTS 生成逻辑应满足以下规则：

- 只有 `entry` 类型中的可朗读文本才进入 TTS。
- `pause` 类型绝不调用 TTS。
- `ja` 在送入日语 TTS 前，可去掉 `[]` 注音，仅保留朗读正文。
- `word.surface` 和 `word.reading` 是否都播放，由播放配置决定，但都必须先经过“可朗读文本”校验。
- 对于清洗后为空、或只剩标点/符号的文本，必须直接跳过，不得送入 TTS。

建议增加统一校验函数，例如：

```ts
function isSpeakableText(text: string): boolean
```

建议校验逻辑至少包括：

- 去除首尾空格。
- 去除 `[]` 注音内容后再次判断。
- 如果文本为空，返回 `false`。
- 如果文本只包含标点、符号、分隔符，返回 `false`。
- 只有包含可实际朗读字符时，才返回 `true`。

### 6.3 服务端兜底要求

即使前端已经过滤，服务端 TTS 接口也应再次校验：

- 收到空字符串，不生成 TTS。
- 收到只含符号的字符串，不生成 TTS。
- 对非法输入直接跳过或返回可识别错误信息。

原因：TTS 稳定性不能只依赖前端，服务端必须做最后一道防线。

## 7. 禁止项

为保证格式统一，以下写法不应再使用：

- 不加前缀、靠行位置猜测语义。
- 词汇行写成一整句自然语言解释。
- 用 `word1:`、`word2:` 这样的序号字段作为最终格式。
- 在同一块里混入 `pause:` 和 `ja:`。
- 把纯符号行当作 `ja:` 句子传给 TTS。
- 把词汇解释继续塞在 `zh:` 里。
- 在同一人物上混用多个姓名读音。

## 8. 最终目标

采用本格式后，应达到以下效果：

- 整理人员可以稳定批量整理轻小说原文。
- 解析器可以用固定规则解析，而不是依赖猜测。
- TTS 不会收到无意义的符号行。
- 前端展示、词汇学习、音频播放三者使用同一份结构化数据。
- 《負けヒロインが多すぎる！》1～8 卷在逐句翻译、注音、人名读音、词条排序上保持一致。
- 后续若要导出 JSON、制作词卡、增加字段，扩展成本更低。

如果后续修改格式，必须同步更新：

- 本说明文档
- 解析器逻辑
- TTS 过滤逻辑
- 示例数据与测试用例
